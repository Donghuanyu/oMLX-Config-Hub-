# Qwen3.6-35B-A3B-4bit 最佳模型设置

## 1. 模型基本信息
* **模型名称**：Qwen3.6-35B-A3B-4bit
* **模型架构类型**：**混合专家模型 (MoE, Mixture of Experts)**
* **架构说明**：
  这是一款混合专家模型，总参数量约为 35B。MoE 架构的核心在于，每次前向传播生成 Token 时并不会激活全部 35B 参数，而是根据输入动态激活其中的约 3B 核心参数（A3B代表激活 3B 参数，Active 3B）。这使得它兼具“35B 级别大模型的表达与逻辑深度”和“3B 稠密模型的极快计算吞吐”。然而，在运行部署上，虽然它的计算开销小，但由于其全部专家权重仍常驻统一内存，它对内存的占用依然与 35B 模型相同（4bit 量化后约为 21GB），故依然依赖高统一内存容量。

---

## 2. 推理加速技术深度解析

为了在 Mac mini 4 (64G) 上实现最高的推理速度与表现，我们对各种加速技术做了深度适配：

### 2.1 DFlash 块扩散投机解码 (当前状态：启用)
* **为什么需要**：尽管 MoE 模型由于仅激活 3B 参数使得计算吞吐极快，但在本地单请求模式下，自回归解码速度的核心限制仍是内存带宽（Memory Bandwidth Bound）。DFlash 通过搭配一个 35B MoE 专属的草稿模型，并行提炼 Token 并在主模型中并发检验，大幅提升了生成效率。
* **优点**：
  * **突破瓶颈**：可为 35B 级别的大模型带来 3-4x 的端到端解码速度提速。
  * **无损验证**：草稿生成的文本经过 35B 主模型的严苛校验，确保逻辑推理和输出质量毫无退化。
* **缺点**：
  * **额外内存占用**：需同时在统一内存中载入 `z-lab/Qwen3.6-35B-A3B-DFlash` 这一草稿模型，需额外占用数GB的统一内存（在 64G 内存上完全能够轻松承受）。
  * **上下文长度受限**：当多轮对话上下文超过草稿模型限制（4096 Token）时，引擎会自动回退到标准自回归解码模式。
  * **单流处理**：适合单人做高负荷的开发工作。

### 2.2 SpecPrefill 投机预填充 (当前状态：禁用)
* **原理**：用一个极小的模型快速计算超长 Prompt 的预填充阶段（Prefill），然后将状态传递给主模型验证。
* **优缺点分析**：
  * **优点**：能够稍微降低长输入（如数万行的代码文件）的首字延迟（TTFT）。
  * **缺点**：与 DFlash 开启时相冲突，且增加额外的统一内存载入开销。由于本模型具有 MoE 的极快计算响应，因此在 64G 统一内存下关闭此项能换取更加稳定且少冲突的运行环境。

### 2.3 Native MTP 原生多Token预测 (当前状态：禁用，无法开启)
* **为什么无法开启**：当前您下载的 4bit 量化版本在转换过程中遗失了原模型声明的 `mtp.*` 权重张量。开启此项会导致 OMLX 报错无法启动。
* **优点**：不需要额外加载任何草稿模型，内存开销极低，可以直接通过主模型中内置的预测头实现约 1.5x 的稳定提速。
* **缺点**：模型文件转换要求高。
* **MTP 启用指南**：
  若想使用 Native MTP 提速，您需要：
  1. 下载官方完整未裁剪的带 MTP 头的 Qwen 官方 HF 原生仓库（例如带有 `MTP` 后缀的版本）。
  2. 使用最新版已合并 PR #990 和 PR #15 补丁的 `mlx-lm` 命令行重新进行 4bit 量化转换。命令如下：
     ```bash
     python -m mlx_lm.convert --hf-path <官方包含MTP权重的HF仓库路径> -q --q-bits 4 --mlx-path <输出路径>
     ```
     这能确保转换生成的 `.safetensors` 文件中包含 `mtp.layers.*` 相关张量，从而可被 OMLX 顺利识别。

## 6. 基准测试结果深度分析与优化方案

### 6.1 核心问题诊断与 MoE 性能评估
1. **MoE 架构的天然吞吐优势（超高基础速度与超低首字延迟）**：
   * **现象**：在 `pp1024` 下，生成速度达到了惊人的 **`167.6 tok/s`**（在 DFlash 加速下）；即便在 DFlash 降级关闭的长上下文段（`pp4096` 至 `pp32768`），生成速度也始终维持在 **`46.5 tok/s` 至 `35.0 tok/s`** 之间。
   * **评估**：相比 27B 稠密模型在 32K 降级后仅有 7.8 tok/s 的速度，Qwen3.6-35B-A3B-4bit 作为 MoE 模型，每次推理仅激活约 3B 核心参数，使其基础自回归解码速度（即使无加速）也有 35.0+ TPS，极具流畅度。
   * **TTFT 优势**：由于 MoE 计算量小，在 `pp32768`（32K）下的首字延迟（TTFT）仅为 **`69.8 秒`**（相比 27B 稠密模型的 284 秒缩短了 75% 以上）。因此，该模型**完全不需要开启 SpecPrefill**，直接使用原生预填充即可获得极佳的响应体验。

2. **DFlash 上限优化（从 4K 提升至 16K）**：
   * **原因**：之前的 `4.20.3 Max Context` 限制在 `4096`，导致 4K 以上的上下文均无法享受到 DFlash 的翻倍提速。由于 64G 内存非常充裕（32K 峰值内存仅 **22.73 GB**），我们建议将 DFlash 限制放宽至 **`16384`** (16K)。
   * **优化预期**：在 16K 以内的对话中，生成速度将获得数倍提升（预计在 4K/8K/16K 上下文均能达到 80~120+ TPS 之间的极速）。当达到 32K 时，由于投机匹配率可能下降，自动回退到 MoE 的原生自回归解码（35.0 TPS），依然能保持极好的流畅体验。

## 8. 更新后的基准测试评估与三次优化方案

### 8.1 更新后的成效分析
1. **🚀 4K 与 8K 上下文生成速度获得惊天暴涨（超级提速）**：
   * **`pp4096`**：Token 生成速度从 **`46.5 tok/s` 飙升至 `160.1 tok/s`**（提升 **244%**），并且 TTFT 从 9.5s 降至 **6.4s**！
   * **`pp8192`**：生成速度从 **`43.6 tok/s` 飙升至 `169.8 tok/s`**（提升 **289%**），基本抹平了中长上下文的解码性能衰减。
   * **评估**：MoE 架构配合 DFlash 在 8K 以内的短/中开发对话中表现出了极为恐怖的性能，基本达到了硬件吞吐的极限。

2. **16K 降级现象与三次微调**：
   * **现象**：`pp16384` 时，生成速度为 **`38.9 tok/s`**，并未获得 DFlash 提速。
   * **原因**：因为 `Max Context` 刚好设在 `16384`。基准测试 `pp16384` 启动时的 Prompt 长度即为 16384，刚好触及并触发了 DFlash 的降级条件。
   * **三次优化手段**：将 `4.20.3 Max Context` 从 `16384` 稍微放宽至 **`20480` (20K)**。因为 16K 时峰值内存仅有 **21.02 GB**，内存毫无压力。放宽到 20K 可以将 16K 上下文完全揽入 DFlash 提速区，使其获得预期的 100+ TPS 超高速。而当多轮对话膨胀到 32K 时，依然可以优雅地降级至 MoE 原生的 **34.7 TPS**，不受任何低匹配率拖累。
