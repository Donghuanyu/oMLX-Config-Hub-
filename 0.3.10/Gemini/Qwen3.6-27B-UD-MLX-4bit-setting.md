# Qwen3.6-27B-UD-MLX-4bit 最佳模型设置

## 1. 模型基本信息
* **模型名称**：Qwen3.6-27B-UD-MLX-4bit
* **模型架构类型**：**稠密型模型 (Dense)**
* **架构说明**：
  这是一款传统的全参数稠密模型，每次前向传播（Token生成）都会激活全部的 27B 参数。与 MoE（混合专家模型）相比，稠密模型在单次生成时需要进行更大量的矩阵乘法计算。在 Apple Silicon 统一内存架构下，稠密模型对内存带宽的压力是恒定且庞大的。因此，这类型模型极其依赖各类推理加速技术来提高每秒输出的 Token 数。

---

## 2. 推理加速技术深度解析

为了压榨 Mac mini 4 (64G) 的硬件潜力，此配置针对多种加速方案进行了定制：

### 2.1 DFlash 块扩散投机解码 (当前状态：启用)
* **为什么需要**：在本地单人开发场景下，大模型的生成速度（解码阶段）主要受限于内存带宽瓶颈（Memory Bandwidth Bound）。DFlash 通过引入一个小型的“草稿模型”（Draft Model）来一次性预测多个候选 Token，然后由主模型在一轮前向传播中并发验证，从而打破了自回归解码一次只能读写一次内存的限制。
* **优点**：
  * **超强加速**：提供无损的 3x 至 4x 的生成速度（TPS）提升。
  * **无损精度**：大模型的验证机制确保了输出的 Token 序列在数学上与直接用 27B 稠密模型生成的结果 100% 一致。
* **缺点**：
  * **额外内存开销**：必须常驻内存加载一个小型的 Draft 模型（通常需多占用几GB内存），不过在 64G 内存上完全够用。
  * **上下文长度限制**：当上下文超过 DFlash 草稿模型所支持的范围（通常为 4096 Token）时，推理引擎会自动降级回退到标准解码模式。
  * **并发受限**：仅支持单请求流，适合单人开发，不适合高并发多用户环境。

### 2.2 SpecPrefill 投机预填充 (当前状态：禁用)
* **原理**：用一个极小的模型快速计算超长 Prompt 的预填充阶段（Prefill），然后将状态传递给主模型验证，旨在降低首字输出延迟（TTFT）。
* **优缺点分析**：
  * **优点**：对于极长的代码文件读取（例如数万 Token），能有效降低开始输出前的“假死”等待感。
  * **缺点**：在短 Prompts 场景下没有加速效果，甚至因为小模型推理的额外消耗导致变慢。此外，它会和 DFlash 冲突，并抢占宝贵的 KV 缓存，在 64G 统一内存下开启 DFlash 已能实现极致的综合体验，因此无需开启 SpecPrefill。

### 2.3 Native MTP 原生多Token预测 (当前状态：禁用，无法开启)
* **为什么无法开启**：您当前下载的 4bit 量化版本在转换过程中遗失了原模型声明的 `mtp.*` 权重张量。开启此项会导致 OMLX 报错无法启动。
* **优点**：不需要额外加载任何草稿模型，内存开销极低，可以直接通过主模型中内置的预测头实现约 1.5x 的稳定提速。
* **缺点**：模型文件转换要求高。
* **MTP 启用指南**：
  若想使用 Native MTP 提速，您需要：
  1. 下载官方完整未裁剪的带 MTP 头的 Qwen 官方 HF 原生仓库（例如 `Qwen/Qwen2.5-Math-7B-Instruct-MTP` 类型的 MTP 原生模型仓库，若 Qwen 3.6 发布了 MTP 版本，请认准带有 `MTP` 后缀的版本）。
  2. 使用最新版已合并 PR #990 和 PR #15 补丁的 `mlx-lm` 命令行重新进行 4bit 量化转换。命令如下：
     ```bash
     python -m mlx_lm.convert --hf-path <官方包含MTP权重的HF仓库路径> -q --q-bits 4 --mlx-path <输出路径>
     ```
     这能确保转换生成的 `.safetensors` 文件中包含 `mtp.layers.*` 相关张量，从而可被 OMLX 顺利识别。

## 6. 基准测试深度分析与优化方案

### 6.1 核心问题诊断
1. **DFlash 提前降级导致的生成速度崩塌 (tg TPS 从 49.7 降至 9.0~7.5)**：
   * **现象**：在 `pp1024` 时，Token 生成速度高达 **49.7 tok/s**，表现出极强的 DFlash 加速效果；但当 Context 达到 `pp4096` 及以上时，速度瞬间断崖式下跌至 **9.0 tok/s**，在 `pp32768`（32K）时更是跌至 **7.8 tok/s**。
   * **原因**：之前的配置将 `4.20.3 Max Context` 限制在 `4096`。一旦 Prompt 或历史对话长度达到 4096，OMLX 引擎就会自动关闭 DFlash，降级为普通解码引擎（即无投机验证的 27B 稠密解码）。
   * **优化建议**：Mac mini 4 M4 (64GB) 内存极其充裕。即使在 32K 上下文下，基准测试的 `Peak Mem` 也仅为 **32.15 GB**，距离 64GB 还有巨大安全余量。因此，**必须将 `4.20.3 Max Context` 上调至 `32768` (或直接留空无限制)**。这样可以确保 DFlash 在整个 32K 窗口中保持活跃，避免超长上下文下的生成卡顿。

2. **超长 Prompt 首字延迟过高 (TTFT 35s - 284s)**：
   * **现象**：`pp4096` 时 TTFT 为 35s，而在 `pp32768` (32K) 时，TTFT 达到了惊人的 **284.7s**（近 5 分钟）。这会给前端工具（Roo Code / Cline）带来严重的“假死/卡死”感，并极易触发工具调用超时。
   * **原因**：默认的预填充 (Prefill) 阶段完全由 27B 稠密模型直接计算，缺乏加速。
   * **优化建议**：对于极长上下文，建议在 OMLX 中开启 **`4.19 SpecPrefill` (投机预填充)**：
     * **Keep Rate**：设置为 `20% — Balanced`（约 3x 预填充提速，无损精度）。
     * **Threshold (tokens)**：设置为 `4096`。这样可以让短 prompt（日常对话）继续使用高精度的标准预填充，而一旦遇到超长文件读取（>4096 tokens），则自动触发 SpecPrefill 投机计算，将 32K 上下文的 TTFT 预估从 **284秒缩短至 ~90秒左右**，极大缓解假死情况。

## 8. 更新后的基准测试评估与二次优化方案

### 8.1 更新后的成效分析
1. **1K 至 16K 上下文生成速度暴涨（重大突破）**：
   * **`pp4096`**：Token 生成速度由之前的 **`9.0 tok/s` 提速至 `37.3 tok/s`**（提速 **314%**），端到端耗时大幅缩短。
   * **`pp8192`**：生成速度由 **`8.9 tok/s` 提速至 `38.1 tok/s`**（提速 **328%**）。
   * **`pp16384`**：生成速度由 **`8.6 tok/s` 提速至 `32.9 tok/s`**（提速 **282%**）。
   * **评估**：这一结果证明了将 `Max Context` 提高到 16K 是极为成功的。在日常开发 95% 的多轮对话场景中，生成速度都获得了接近 4 倍的无损暴涨！

2. **32K 上下文的性能衰退现象（新发现的瓶颈）**：
   * **`pp32768`**：开启 DFlash 后，`tg TPS` 降至 **`5.5 tok/s`**（甚至低于未加速前的 `7.8 tok/s`），且 `TTFT` 增至 **`402.7秒`**。
   * **原因分析**：DFlash 采用的滑动注意力机制（Window Size 1024 / Sink 64）在超长上下文（32K）下匹配度（Verify Success Rate）大幅降低。因为草稿模型预测的 Token 大量被主模型拒收，导致了大量的冗余验证和无效计算，开销反而大于普通自回归解码。
   * **二次优化手段**：将 `4.20.3 Max Context` 从 `32768` 细调至 **`16384`**。这样既能完美保留 16K 以内接近 40 TPS 的极速，又能在上下文膨胀到 32K 时，自动且优雅地回退到标准的普通自回归解码模式（获得较优的 `7.8 tok/s`，且避免了投机验证的 TTFT 额外开销）。

### 8.2 关于首字延迟（TTFT）的补充说明
* 尽管投机预填充（SpecPrefill）已按建议开启，但 32K 下的 prefill 速度（`81.4 tok/s`）由于统一内存带宽在 32K 超长 Key-Value 张量运算下的硬性开销，仍然有一定耗时。
* **建议**：在 Roo Code/Cline 侧，当处理多文件和巨型上下文时，依然建议借助 [前端提示词规则](file:///Users/dong/Documents/intellij_workspace/native-ai-setting/0.3.10/Gemini/Qwen3.6-27B-UD-MLX-4bit-setting.md#L89) 分段读取文件，避免单次直接拉满 32K，从而兼顾首字响应时间。`16384` 限制 of DFlash 能够确保 16K 内的开发流畅度拉满。
